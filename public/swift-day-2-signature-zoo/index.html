Swift (Day 2) - Signature Zoo

06 Nov, 2024

I alluded last time to the density of keywords and language primitives in Swift being quite ample (reminds me of my experience learning Rust, and my [extensive] time with Scala). As I mess around with the vanilla tutorials above, I’m diving into some definitions (cmd + Click in Xcode), which are quite illuminating. Let’s take a few and talk about them.

View: the very first protocol you encounter when working with SwiftUI:

Let’s go through some of this:

  • public: This is just access control; apparently there’s also an open keyword, where the difference is that open classes can be subclassed and overridden from other modules, whereas public classes can’t be. Protocols can’t be open, since the whole point of a protocol is to be implemented (or “protocol conformance”, as seems to be the parlance).
    • There’s a lot of access control stuff - for now, we just remember that modules are a single unit, and packages are a group of modules.
  • associatedtype: this is generics, like in Scala or Golang, and is later referred to by the Self.Body type. This associated type is a placeholder, and must be specified by implementing types. The Body itself must conform to the View protocol, which seems recursive, but just means that some custom struct implementing a View can then have a body composed of other predefined views, like Text or whatever.
    • This is different than typealias, which is just a different name for a type, but has nothing to do with conforming a protocol.
  • @MainActor: this is an attribute, like @ViewBuilder and @preconcurrency. But things get weird for me here. Some of these seem purely informational, like @preconcurrency (see below). But some are loaded with methods themselves, like this and @ViewBuilder.
    • What it actually does is simpler, as seen here - it marks code to run on a singleton actor that’s basically the main thread. Things like UI changes need to be run on the main thread, to avoid race conditions on crucial state updates.
    • It’s just not terribly clear to me yet the relationship between these different kinds of primitives in Swift. It’s a lot of overhead.
  • We’ll get into @ViewBuilder below, but it’s even more of a doozy.
  • The code in all of this is not terribly helpful to read, because so much of it is closed-source.

ViewBuilder: the struct behind the @ViewBuilder _attribute:

Woof. This is even more furious. Let’s start.

  • First and foremost - this is closed-source code! So the reason this looks like a protocol is because we can’t see the actual code. Bummer.
  • This struct has two extension blocks. They have two different @available attributes annotated on them, e.g. they’re available for different platforms. So that makes sense to me why they’d be separate.
    • The argument I’ve heard for totally different blocks despite no differences other than contained methods is just modularity / separation of concerns. That resonates okay with me.
  • @resultBuilder: let’s go through some questions.
    • “What does this do?” - well, apparently it enables DSLs like Views, e.g. most of the underpinnings of SwiftUI. You can plop things like Text and Image into an HStack without much fuss, and they compose (into a TupleView struct, apparently) just fine because of how the @resultBuilder maps elements to its methods above.
    • “Why is this lower-case?” - well, it’s closed-source, and not even the stubs are visible. That’s probably why.
    • “Why use this?” - I dunno, people like declarative DSLs that let them get up and running quickly? Frankly, I don’t love the taste of this so far, especially as I read about implementing things like buildEither just to include if-statements in your closure. It’s the curse of all DSLs - people demand more and more expressiveness (or developers get thirsty) until the differences between the DSL and the original language seem somewhat minimal to me. That’s why I resonate with things like Pulumi over OpenTofu / HCL - just use Golang, even if it’s a bit kludgy. But then again, I’m a Golang developer. But then again then again, I used Scala extensively for years and never enjoyed the DSL-y aspects of it (including SBT) - it just felt like learning a new, leaky-as-hell abstraction. Okay, I digress.
    • “Can I learn more concretely here, John?” - sure, read some of the many articles dedicated to this. I make no promises of education here, lad.

  • buildExpression(_ content: Content): two notes:
    • Content is just another generic. Fine. Apparently Swift allows predicates on generics, e.g. where Content : View, e.g. “where our generic Content must fulfill the View protocol”.
    • (_ content: Content) is weird because it has two argument names, which is again our internal and external variable names. Here, **_ is just a placeholder, e.g. “no external name required”.

And finally…

WRAP THAT SIGNATURE

Jeesus! Okay, well, I found this SwiftLee article helpful once again. Here’s this in plain English, spoken from the perspective of the function:

  1. I, buildBlock, will accept many many different Content types.
  2. However, they all must implement the View protocol.
  3. But pass as many as you like! HStack, Text, Image, I don’t care!

This function is rather simple, but the language concepts embedded into it are pretty rich. It’s also just keyword-heavy. But I think this is the way that things like HStack and VStack accept totally different things like Image and Text. They all have different types, but they’re all implementing View, so we pass in a bunch and each one, repeatedly (wink) gets handled by buildBlock.

CONCLUSION: Swift was designed by experienced developers who are NOT grouchy old-school devs like the developers of Golang.

I think I’d like these blogs to be at least slightly helpful, in addition to being my educational vehicle. So I think I’ll just read all the comments and attributes and distill them into a little memory cheatsheet. Till then.
